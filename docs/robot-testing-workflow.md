# Robot Testing Workflow

Этот runbook фиксирует обязательный порядок действий для ad-hoc тестирования роботов OsEngine в этом репозитории.

## Когда использовать

Использовать для запросов вида:
- "протестируй робота";
- "прогони стратегию в Tester";
- "собери статистику по роботу";
- "сделай отчет по бэктесту".

## 1. Согласование перед запуском

Перед каждым прогоном показать пользователю нумерованный список настроек.

Минимальный шаблон:

1. Робот
2. Режим робота
3. Инструмент
4. Класс инструмента
5. Источник данных Tester
6. Таймфрейм
7. Период теста
8. Режим тестера / Fast / тип данных
9. Стартовый портфель / комиссия
10. Параметры стратегии
11. Какие статистики и артефакты будут собраны

Запускать можно только после явного подтверждения пользователя.

## 2. Выполнение

Предпочтительный порядок:
1. Проверить входные пути и данные.
2. Предпочесть code-driven или in-process run вместо хрупкого GUI-scraping.
3. Если без GUI нельзя, использовать `project/OsEngine.TesterAutomation`.
4. Во время прогона следить за логами, завершением теста и критическими ошибками.

## 3. Что собирать после прогона

Минимальный обязательный набор:
- статус;
- фактические настройки;
- длительность;
- период теста;
- стартовый и итоговый портфель;
- число сделок и разбиение по направлениям;
- прибыльность, winrate, drawdown, PF, Sharpe, Recovery;
- long/short блоки;
- state counts;
- error/warning logs;
- путь к сырому артефакту и к финальному опубликованному отчету.

## 4. Финальный отчет

Финальный отчет публикуется в `reports/tests/` и состоит из двух файлов:
- JSON с полной структурой данных;
- Markdown со сводкой для чтения.

Схема имени:

```text
yyyy-MM-dd_HH-mm-ss__<Robot>__<Ticker>.json
yyyy-MM-dd_HH-mm-ss__<Robot>__<Ticker>.md
```

Примеры:
- `2026-03-12_18-51-43__BreakEma__Сбербанк.json`
- `2026-03-12_18-51-43__BreakEma__Сбербанк.md`

## 5. Публикация отчета

Если сырой структурированный JSON уже собран, опубликовать финальный отчет так:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File tools/Publish-RobotTestReport.ps1 -InputJsonPath reports\raw_run.json
```

Скрипт:
- создает `reports/tests/`, если каталога нет;
- нормализует имя файла;
- пишет финальный JSON;
- пишет Markdown companion;
- возвращает пути к обоим артефактам.

## 6. Что сообщать пользователю

После прогона обязательно сообщить:
- фактические настройки;
- путь к финальному отчету;
- ключевой финансовый итог;
- основные trade-метрики;
- ошибки/отклонения/неполноту данных, если они были.
