# Тестирование роботов OsEngine

Следовать этому документу как обязательному workflow для всех запросов на тестирование / бэктест / "прогони робота" в этом репозитории.

## 0. Назначение

Этот skill покрывает:
- согласование настроек прогона с пользователем;
- запуск теста через наиболее стабильный доступный путь;
- сбор полной статистики и ошибок;
- выпуск итогового отчета в `reports/tests/`.

Если задача включает изменение кода робота, одновременно применять:
- `$HOME/.codex-personal/skills/osengine-robot-authoring/SKILL.md`
- этот skill для части run/reporting.

## 1. Обязательный pre-run checklist

Перед каждым прогоном MUST вывести пользователю нумерованный список настроек и попросить:
- подтвердить все сразу;
- или изменить конкретные пункты по номерам.

Минимальный список для подтверждения:
1. робот;
2. режим робота (`On` / иной фактический режим);
3. инструмент / файл бумаги;
4. класс инструмента;
5. источник данных Tester и тип источника (`Folder`, `Set`, и т.д.);
6. таймфрейм;
7. период теста;
8. режим тестера (`Candle` / `Tick` / `MarketDepth`), `Fast` и иные важные runtime-переключатели;
9. стартовый портфель / комиссия / иные условия капитала, если они участвуют в результате;
10. параметры стратегии;
11. что именно будет собрано после прогона: сделки, PnL, drawdown, long/short, ошибки, логи, итоговые артефакты.

## 2. Правила согласования

- Нельзя запускать тест без явного подтверждения пользователя, кроме случаев, когда он уже подтвердил эти пункты в текущем диалоге.
- Фразы вида `подтверждаю всё`, `ок, тестим`, `да, запускай` считаются подтверждением всего списка.
- Если буквальные default-параметры дают заведомо пустой прогон (пример: `Regime = Off`), MUST явно предупредить до запуска и предложить исправление.
- Если часть настроек можно разумно вывести из локального контекста без риска, допустимо заполнить их автоматически, но всё равно показать пользователю перед стартом.

## 3. Правила запуска

- Сначала проверить существование входных данных и путей.
- Предпочитать наиболее стабильный путь запуска:
  1. in-process / code-driven Tester run;
  2. существующий automation harness;
  3. GUI automation только когда без GUI нельзя воспроизвести нужный сценарий.
- Во время прогона фиксировать фактические настройки, а не только желаемые.
- Следить минимум за:
  - основным процессом;
  - логами `Engine\\Log\\*.txt`;
  - моментом завершения теста;
  - неожиданными error/modal состояниями.

## 4. Обязательная post-run статистика

После завершения MUST собрать и сохранить:
- статус прогона;
- фактический период теста;
- длительность;
- стартовый и итоговый портфель;
- число всех, закрытых и открытых позиций;
- long/short разбивку;
- число прибыльных / убыточных / нулевых сделок;
- winrate;
- net profit, gross profit, gross loss, commission;
- первую и последнюю сделку;
- state counts (`Done`, `OpeningFail`, и т.д.);
- ключевые встроенные метрики OsEngine:
  - `Чистая прибыль`
  - `Чистая прибыль, %`
  - `Количество сделок`
  - `Среднее время в позиции`
  - `Шарп`
  - `Profit Factor`
  - `Recovery`
  - `Максимальная просадка, %`
- long и short статистику отдельными блоками;
- ошибки и предупреждения из логов;
- путь к исходным артефактам, если они есть.

Если доступны дополнительные показатели, их тоже сохранять в machine-readable JSON.

## 5. Итоговый отчет

После каждого прогона MUST выпустить финальный отчет-пакет в `reports/tests/`.

Обязательный формат:
- machine-readable JSON;
- human-readable Markdown.

Обязательная схема имени файла:
- `yyyy-MM-dd_HH-mm-ss__<Robot>__<Ticker>.json`
- `yyyy-MM-dd_HH-mm-ss__<Robot>__<Ticker>.md`

Где:
- timestamp берется из фактического времени запуска;
- `Robot` и `Ticker` нормализуются для файловой системы;
- тикер берется без расширения `.txt`, если оно есть.

Raw/debug JSON может оставаться в `reports/` или другом месте, но финальный опубликованный отчет MUST лежать в `reports/tests/`.

Для публикации финального отчета использовать:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File tools/Publish-RobotTestReport.ps1 -InputJsonPath <raw-report.json>
```

## 6. Ответ пользователю после прогона

В ответе MUST указать:
- какие настройки реально использовались;
- путь к финальному отчету в `reports/tests/`;
- ключевой финансовый итог;
- базовую trade-статистику;
- были ли ошибки / warning / отклонения от согласованных настроек.

Если промежуточные попытки были неуспешны, это нужно кратко зафиксировать, но ссылаться в первую очередь на финальный успешный отчет.

## 7. Опорные файлы

- runbook: `docs/robot-testing-workflow.md`
- publisher script: `tools/Publish-RobotTestReport.ps1`
- final reports directory: `reports/tests/`
